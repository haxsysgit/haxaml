# Haxaml 0.6.x Roadmap

This roadmap defines the path from `0.6.2` to `0.7.0`.

The purpose of the `0.6.x` line is not to jump straight into the full setup and
agent-control system. The purpose is to make Haxaml:

- more consistent
- less noisy
- smarter with FRAME
- cheaper to run on long projects

Short version:

> `0.6.x` is the hardening and intelligence phase.  
> `0.7.0` is the setup and control phase.

---

## What The 0.6 Series Is Trying To Achieve

By the time `0.6.x` is complete, Haxaml should feel like this:

- the public story is clear and consistent
- the lifecycle is strict, but not chatty
- FRAME carries more meaning without more clutter
- long projects stay usable without context bloat
- agents do less unnecessary rereading
- the project is ready for the full setup/control layer in `0.7.0`

The main theme for the whole series is:

> Make Haxaml smaller to read, smarter to reason with, and harder to drift from.

---

## Release Shape

Each `0.6.x` release should answer five simple questions:

1. What problem is this release fixing?
2. What will feel better to the user or agent after it lands?
3. Which parts of the package are most likely to change?
4. What should tests prove before the release is done?
5. What is intentionally postponed to the next release?

That keeps the series practical instead of turning into a pile of good ideas.

---

## 0.6.2 - Full Unification And Consistency

### Goal

Make Haxaml say the same thing everywhere.

This is the cleanup and alignment release.

### Why This Release Matters

Right now the package is strong, but parts of it still sound like different eras
of Haxaml:

- older lifecycle wording still appears in some places
- `prebuild` is the intended front door, but older paths still show up publicly
- docs, help text, payload wording, and guidance messages are not always aligned

That creates avoidable confusion.

### What Should Change

- Make `haxaml_prebuild` the obvious public front door for governed work.
- Align all lifecycle wording around the same preferred flow:

```text
about -> guidance -> prebuild -> context_pack -> build -> verify -> record -> expect_sync
```

- Reduce visible confusion from older public traces of:
  - `haxaml_session_start`
  - `haxaml_session_plan`
  - deprecated wrappers
- Align:
  - README
  - learn docs
  - MCP docs
  - tool reference docs
  - changelog
  - CLI help text
  - validation hints
  - benchmark wording
  - test expectations
- Make error guidance point to the same recommended flow.

### What The User Should Feel

- the package feels easier to understand
- there is one main way to use governed flow
- the docs and tool messages stop contradicting each other

### Likely Code Areas

- `haxaml/mcp/tools_prebuild.py`
- `haxaml/mcp/tools_lifecycle.py`
- `haxaml/mcp/tools_frame.py`
- `haxaml/mcp/response_helpers.py`
- `haxaml/mcp/policy_helpers.py`
- docs and help text

### Tests Should Prove

- `prebuild` is visible and documented as the preferred path
- lifecycle help text and payloads match the same public flow
- stale public traces of alternate flows are removed

### Not The Goal Yet

- no big lifecycle redesign
- no inference engine
- no cache or UI work yet

## 0.6.3 - Lifecycle Optimization And Noise Reduction

### Goal

Make governed work cheaper and less noisy.

### Why This Release Matters

Even when the lifecycle is correct, it can still feel heavier than it should:

- repeated explanations
- repeated reads
- repeated context pack usage
- too much text when the next step is already obvious

This release is about making Haxaml feel more disciplined and less wasteful.

### What Should Change

- Reduce repetitive lifecycle payloads and repeated explanatory text.
- Tighten the “one context pack per task unless needed” policy.
- Make refresh reasons clearer and more consistent.
- Improve utility-mode boundaries so side tasks stay outside governed state cleanly.
- Reduce unnecessary rereads in the recommended path where that can be done safely.
- Update benchmark profiles to reflect the real preferred flow, not older session-heavy public behavior.
- Add earlier drift checks before session completion can be recorded.
- Start replacing abstract or overly technical user-facing terms with more contextual wording.

### What The User Should Feel

- fewer unnecessary lifecycle steps
- less repeated guidance text
- cleaner governed runs
- clearer boundaries between real governed work and utility work

### Likely Code Areas

- `haxaml/mcp/tools_lifecycle.py`
- `haxaml/mcp/lifecycle_helpers.py`
- `haxaml/mcp/policy_helpers.py`
- `haxaml/mcp/response_helpers.py`
- benchmark tests and lifecycle tests

### Tests Should Prove

- repeat context pack calls are disciplined
- utility tasks stay outside governed state
- repeated gate failures do not produce noisy retry loops
- lifecycle guidance remains strict while becoming shorter

### Not The Goal Yet

- no major new data model
- no long-project compaction overhaul yet
- no UI yet

## 0.6.4 - Deterministic Drift And Consistency

### Goal

Make Haxaml smarter by deriving deterministic consistency signals from existing FRAME.

### Why This Release Matters

This release strengthens drift detection and current-state summaries without adding a speculative inference layer.

### What Should Change

- Strengthen deterministic checks in `validator`, `reconcile`, `doctor`, `health`, and `prebuild`.
- Derive advisory run and phase coherence warnings from `expect.yaml` and `acts.yaml`.
- Detect stale active-task, open-session, and verification-discipline mismatches.
- Produce a compact derived progress summary such as `on_track`, `blocked`, `sync_pending`, or `stale_state`.
- Keep new contradictions advisory-first unless they fall into already-established blocker categories.

### Important Design Rule

Do **not** add a permanent top-level `outcomes.yaml`.
Do **not** add project-kind inference to Haxaml.

### What The User Should Feel

- FRAME becomes smarter without becoming bigger
- contradictions are easier to explain
- readiness and health surfaces carry better current-state summaries
- the project status feels clearer without adding more files

### Likely Code Areas

- `haxaml/validator.py`
- `haxaml/reconcile.py`
- `haxaml/mcp/tools_frame.py`
- `haxaml/mcp/tools_prebuild.py`

### Tests Should Prove

- phase/run mismatch warnings are advisory
- stale session and stale active-task findings surface consistently
- doctor/reconcile explain contradictions more clearly
- progress reporting can be produced from FRAME without a new top-level file

### Not The Goal Yet

- no full UI yet
- no setup/skill system yet
- no broad persistence redesign yet

## 0.6.5 - Compatibility Purge And Canonicalization

### Goal

Remove compatibility architecture and make Haxaml canonical-only.

### Why This Release Matters

Compatibility layers made sense when the public flow was still settling. They are now extra surface area, extra docs burden, and extra ways for the agent to enter the wrong path.

### What Should Change

- Remove deprecated or alternate public MCP lifecycle surfaces.
- Keep `haxaml_prebuild` as the only public governed session-entry tool.
- Remove compatibility wrappers and compatibility re-export modules.
- Remove legacy FRAME file names and non-canonical path fallbacks.
- Remove `standard` pack alias and keep only `minimal`, `balanced`, and `full`.
- Rewrite docs, tests, and MCP references around the canonical flow only.

- there is only one governed story to learn
- compatibility wording disappears from the repo
- FRAME locations and names are predictable
- agents have fewer wrong entrypoints to choose from

### Likely Code Areas

- `haxaml/mcp_server.py`
- `haxaml/mcp/tools_lifecycle.py`
- `haxaml/mcp/tools_frame.py`
- `haxaml/mcp/resources.py`
- `haxaml/cli.py`
- `haxaml/paths.py`
- docs and MCP help text

### Tests Should Prove

- removed tools and resources are no longer registered
- removed CLI commands are gone
- canonical `.haxaml/*.yaml` is the only supported FRAME location
- only `minimal|balanced|full` are accepted context packs
- docs and tests no longer reference removed compatibility paths

### Not The Goal Yet

- no state compaction overhaul yet
- no UI-first work yet

## 0.6.6 - Tiered Acts Memory And Guided Retrieval

### Goal

Keep long-running projects governable without letting `acts.yaml` become a giant history dump.

### Why This Release Matters

Haxaml becomes much less useful if long projects force every agent to drag around too much old state.

### What Should Change

- Add deterministic compaction rules for `acts.yaml`.
- Keep recent runs in full detail.
- Collapse older runs into useful summaries.
- Add decision compression so important lessons survive when old runs are compacted.
- Separate hot recent state from cold historical state.
- Make context packs and exports prefer active recent state instead of raw historical growth.

### What The User Should Feel

- long projects stay readable
- agents can regain orientation faster
- the project diary stays useful instead of bloated

### Likely Code Areas

- `haxaml/context.py`
- `haxaml/state_manager.py`
- `haxaml/runner.py`
- `haxaml/mcp/tools_ops.py`
- `acts` schema if additional compaction metadata is needed

### Tests Should Prove

- compaction is deterministic
- recent state is preserved correctly
- summaries keep enough useful meaning
- context generation prefers active state after compaction

### Not The Goal Yet

- no full freshness engine yet
- no UI-first work yet

## 0.6.7 - Incremental Read Policy And First Local Interface

### Goal

Stop rereading stable files unnecessarily, and add the first human-facing review layer on top of FRAME.

### Why This Release Matters

Haxaml should not force agents to reread the whole manual when only one page changed.

And by this point, the project should benefit from a small interface that lets humans review project health and progress without replacing the file-based model.

### What Should Change

- Add smarter incremental read policy.
- Teach Haxaml the difference between:
  - stable project facts
  - active execution state
  - rules that may need refresh
  - recently changed sections
- Add freshness markers and invalidation logic.
- Prefer section-level refresh over file-level reread where possible.
- Add refresh diff summaries so agents can see what changed since the last pack.
- Improve long-session caching behavior.
- Add the first local interface on top of FRAME as a thin layer over the files.

### The First Interface Should Show

- project health
- readiness
- facts/rules/acts/expect editing
- progress derived from existing FRAME files
- drift and warning visibility

### Important Design Rule

The interface should sit on top of the current files.

It should **not** become a new storage system.

### What The User Should Feel

- fewer wasted reads
- smaller context packs
- better continuity on long tasks
- easier human review of what Haxaml thinks is happening

### Likely Code Areas

- `haxaml/context.py`
- `haxaml/frame_model.py`
- `haxaml/mcp/lifecycle_helpers.py`
- `haxaml/mcp/tools_prebuild.py`
- `haxaml/mcp/tools_frame.py`
- a new local UI layer if kept in this repo

### Tests Should Prove

- stable files are not reread without reason
- refresh summaries correctly describe what changed
- context packs shrink when only active state changed
- the interface reflects real file state correctly

### Not The Goal Yet

- no full setup/skill control system yet
- no agent adapter generation yet

## What 0.7.0 Then Becomes

After `0.6.x` is hardened, `0.7.0` becomes the real setup and agent control milestone.

That is where Haxaml should deliver the architectural enforcement layer:

- built-in Haxaml Skill
- generated adapters for major agents
- `haxaml setup`
- project-scope and user-scope setup paths
- managed sections in native instruction files
- setup doctor
- stronger repo-level enforcement so Haxaml is harder to forget across sessions and tools

Simple meaning:

> `0.7.0` is where Haxaml stops being something the agent should use and becomes how the repo is wired.

---

## Recommended Sequence

Short form:

- `0.6.2` = unify the story
- `0.6.3` = reduce lifecycle noise
- `0.6.4` = deterministic drift and consistency
- `0.6.5` = compatibility purge and canonicalization
- `0.6.6` = tiered acts memory and guided retrieval
- `0.6.7` = incremental reads and first local interface
- `0.7.0` = full setup and agent control system

---

## Coding Approach For The Whole Series

The safest and cleanest implementation pattern for this series is:

1. refine the data shape or contract first
2. add or improve helper-layer logic
3. expose the behavior through MCP or CLI surfaces
4. align docs and tests immediately after

That keeps the project from turning into endpoint-level sprawl.

It also matches how Haxaml is already strongest:

- normalized model
- helper-driven logic
- thin public tool surfaces
- durable file-based state
