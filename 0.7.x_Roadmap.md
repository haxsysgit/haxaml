# Haxaml `0.7.x` Roadmap

`0.7.x` is the setup and workflow accommodation line.

The canonical product vision lives in [goals.md](./goals.md). This roadmap is only about sequencing the `0.7.x` line.

`0.6.x` made Haxaml coherent as a governed workflow and MCP runtime. `0.7.0` already shipped the setup foundation. The rest of `0.7.x` is about moving Haxaml from "installed correctly" to "active inside real agent workflows" using workflow-native integration points that official vendor docs now expose.

Haxaml is the governance spine for agent work, not an AI memory product. Memory, archive retrieval, and diary state remain supporting systems for governed execution, not the headline. The `0.7.x` line therefore focuses on getting real agents into the existing governed lifecycle:

`about -> guidance -> prebuild -> context_pack -> verify -> record -> expect_sync`

Roadmap policy for this line:

- Automate only stable, documented, file-backed, config-backed, agent-backed, or hook-backed integration points.
- Keep UI-only or weakly documented workflow behavior as guidance-only.
- Prefer tool-specific workflow accommodation over inventing parallel Haxaml workflow runtimes.
- Use `0.7.x` to make stable gates real where they already have a clear contract. Reserve broader enforcement and migration-heavy blocking for `0.8.0`.

## `0.7.x` Product Position

`0.7.x` has one job: make Haxaml usable where agents already live without weakening the README contract.

This means several concrete shifts:

- setup is already shipped in `0.7.0`; the remaining work is workflow accommodation depth
- setup must feel like a scaffold command with explicit options, previews, and boringly reliable target selection
- workflow should move from reactive tool-calling toward proactive lifecycle steering
- materials gathering must become more visible in governed flow, especially owner-provided dependencies and `materials_needed`
- context-budget planning is product usability work, not just token-efficiency work
- diary and handoff state must synthesize decisions and failures, not just accumulate raw Acts history
- setup reruns and target selection must become boringly safe before the line is considered done
- Acts/archive compaction must stop putting old match replays in the active loadout and must not require manual cleanup to stay usable
- RAG research is a later-line concern, not a `0.7.x` ship target

Semantic retrieval remains strategically important, but it does not belong in `0.7.x` as a promised default feature. It stays on the pre-`1.0` path as a research-first track in `0.9.0`, where it can be evaluated alongside RAG, embeddings, and retrieval-acceleration options.

## `0.7.0` Delivered Foundation

`0.7.0` is already the delivered setup foundation for the line.

Delivered direction:

- `haxaml setup` as the main onboarding command
- `haxaml setup doctor` for install audit, drift detection, and manual follow-up reporting
- `haxaml setup print` for inspect-before-write output
- one shared target registry for setup behavior and docs
- one canonical Haxaml operating contract rendered into tool-specific setup outputs
- explicit project-scope versus user-scope handling

`0.7.0` solved "how do I install Haxaml into this environment?" The remaining `0.7.x` releases solve "how does that environment keep Haxaml active during real work?"

## `0.7.1` - Workflow Accommodation Foundation

Goal: extend setup from static onboarding assets into workflow-native outputs.

Planned work:

- add roadmap support for workflow-native setup outputs, not just instructions, skills, agents, and MCP
- extend `haxaml setup` with a `workflow` asset class so setup can install or print workflow-native assets alongside `frame`, `instructions`, `skills`, `agents`, and `mcp`
- add a shell-safe workflow guard CLI integration point for hooks, custom agents, and remote/background environments
- keep the workflow guard explicit as a machine-friendly entrypoint with exit codes and `text`/`json` output
- keep hooks and remote agents on CLI-safe helpers rather than routing them through a new MCP tool

Primary targets for `0.7.1`:

- Claude Code hooks
- GitHub Copilot custom agents
- OpenCode skills and agents
- Junie portable skills
- Codex config-backed workflow assets

## `0.7.2` - Provider Workflow Depth

Goal: deepen target-specific workflow accommodation where vendor docs are now strong enough.

Planned work:

- Claude Code: hook matchers, steering, and selective gating patterns
- GitHub Copilot: `.github/agents`, `.github/skills`, and repository-vs-user MCP precedence
- Cursor: background-agent setup assets and `.cursor/environment.json` guidance where the workflow integration point is file-backed and documented
- OpenCode: skill-first and agent-first workflow integration with per-agent MCP/tool enablement
- Junie: portable skill-first workflow integration and subagent-aware handoff
- keep unsupported or UI-only workflow integration points guidance-only

## `0.7.3` - Workflow Doctor And Drift Audit

Goal: make workflow accommodation inspectable, not just installable.

Planned work:

- keep `haxaml setup doctor` as the single advisory audit entrypoint for setup-managed and workflow-managed files
- make workflow doctor output provider-aware so managed files are described as hook scripts, hook config, background environments, custom agents, adapter files, local runners, CI runners, and workflow MCP/config entrypoints instead of generic manifest entries
- enrich doctor JSON with optional workflow metadata while keeping the top-level `installed / missing / drifted / manual_actions / message` report shape stable
- keep scope limited to the workflow-capable targets already shipped in `0.7.2`
- keep `workflow check` as the stricter readiness path and do not add a failing doctor mode in `0.7.3`

## `0.7.4` - Interactive Setup And Skill Overhaul

Goal: turn setup into a confident onboarding UX instead of a thin file generator.

Planned work:

- make `haxaml setup` interactive by default in a real TTY while keeping `--format json` and `--non-interactive` machine-friendly
- treat target detection as advisory, not authoritative: strong tool-specific files can preselect targets, but shared weak signals like `AGENTS.md` must not silently resolve multiple providers
- replace the thin generated `SKILL.md` checklist with valid YAML-frontmatter skills that actually encode lifecycle triggers, success criteria, fallback behavior, escalation rules, and provider-aware examples
- add narrow safe merges for known MCP config shapes so Haxaml can update only `mcpServers.haxaml` or `[mcp_servers.haxaml]` without clobbering unrelated user config
- replace count-only setup output with exact paths plus short previews so users can see what will change before Haxaml writes anything

`0.7.4` established the direction, but real-project testing showed that setup is still too thin, too weakly structured, and too easy to mistrust. `0.7.5` is therefore not a cosmetic follow-up. It is the release that has to make setup feel intentional and make active FRAME state stay compact enough for real use.

## Observed `0.7.5` Correction Targets

Real-project testing after `0.7.4` exposed two specific failures that must shape the next release:

- hot `acts.yaml` still grows into long replay history, which makes active context noisy and forces users to remember to run archive commands manually
- the current interactive setup flow is too thin and too raw compared to scaffold tools like `create-vue`, and target selection can still fall back to `generic` in situations where the user expected an explicit provider-specific setup

The release should treat these as product trust failures, not polish-only issues.

## `0.7.5` - Materials, Context Discipline, And Handoff Quality

Goal: make governed flow outputs easier to act on after setup itself becomes trustworthy, while closing the state/context issues found in the 2026-05-17 audit.

Planned work:
- reshape `haxaml setup` around a scaffold-style terminal flow with spacing, color, grouped options, multi-select feature picks, a review screen, and concrete next steps instead of a thin prompt chain
- keep the setup engine in core Haxaml; if a richer terminal frontend would add unacceptable dependency or runtime overhead, ship that frontend as an optional `uvx`-runnable setup package that calls the same planning/apply engine rather than creating a second setup logic stack
- make setup option selection feature-first rather than question-first: target scope, adoption mode, provider targets, installable capabilities, workflow adapters, safety defaults, and rerun behavior should be explicit first-class choices
- keep target detection advisory only; explicit provider selections must survive planning and apply unchanged and must never degrade to `generic`
- make setup review output show exact files, targets, capabilities, merge behavior, and follow-up actions before writes happen
- make setup reruns preserve active FRAME state by default and limit writes to create-missing or safe-repair behavior unless a future explicit reset mode is requested
- improve the terminal UX to the level of a serious scaffold tool, not a basic questionnaire; the benchmark is convenience and clarity, not a flashy full-screen UI

- make `materials_needed` and owner-provided dependencies more prominent in generated guidance and workflow outputs
- improve task-aware, risk-aware, and model-aware context budget planning

Design intent for `0.7.5`:

- `expect.yaml` remains the planned end, expected runs, checkpoints, and finish line
- `acts.yaml` records what actually happened, but hot state should carry only the active decision trail and the shortest useful run history
- `map.yaml` keeps ownership and impact boundaries so agents can see where the next run is expected to operate
- archive is for cold evidence and retrieval, not for pushing an oversized replay log back into active context
- setup is a trust surface: if provider selection, previews, and rerun safety are weak, adoption will stay weak no matter how strong the governed runtime becomes

## Important Planned Public Integration Points

These public integration points should be explicit in the roadmap so implementation intent is not ambiguous:

- `haxaml setup --only workflow`
- workflow as a first-class setup asset category alongside `frame`, `instructions`, `skills`, `agents`, and `mcp`
- a shell-safe workflow guard CLI integration point for hooks, custom agents, and remote/background environments
- machine-readable workflow guard output in `json` as well as human-readable `text`
- exit-code failure for non-interactive workflow checks
- no new workflow MCP tool in `0.7.x`

`0.7.x` workflow accommodation should route tool-specific workflow assets into the existing governed lifecycle. Hooks, CI jobs, custom agents, and remote agents need CLI-safe helpers that call into Haxaml's established flow, not a second parallel workflow API.

## Out Of Scope

The `0.7.x` line stays disciplined about what workflow accommodation is not:

- no dashboard expansion into a workflow control plane
- no secret manager
- no UI automation for vendor settings dialogs
- no undocumented private config-path guessing
- no new memory-product positioning
- no default semantic-retrieval rollout

## Acceptance Checks

Roadmap consistency:

- `0.7.x_Roadmap.md` and `v1.0_Roadmap.md` must both state that `0.7.0` setup is already shipped
- `v1.0_Roadmap.md` must link to `0.7.x_Roadmap.md`, not `0.7.x.md`
- release numbering must stay monotonic across `0.7.0`, `0.7.1` through `0.7.5`, `0.8.0`, `0.9.0`, and `1.0.0`

Source alignment:

- every workflow-native automation claim must remain tied to a current official source
- unsupported or UI-only workflow integration points must stay guidance-only
- tool-specific workflow integration should prefer vendor-supported files, hooks, agents, configs, and CLI integration points over speculative abstractions

`0.7.5` release checks:

- explicit provider selections remain explicit through setup planning, manifest output, and applied files; they are not silently downgraded to `generic`
- setup review output shows exact targets, capabilities, files, and follow-up actions before write
- setup reruns preserve populated FRAME files by default unless the user explicitly chooses a destructive reset mode
- hot `acts.yaml` compacts automatically before it becomes token-clogging, and the compaction policy is byte-budget aware rather than count-only
- hot `completed_tasks` remains bounded and summarized instead of growing without limit
- archived run/session/verification history stays retrievable after compaction without reloading full replay history into active context
- blocking questions and `materials_needed` can stop context pack/build instead of being buried in advisory output

Product-language checks:

- Haxaml should be described primarily as workflow governance, not as memory tooling
- materials gathering and context budgeting should be treated as workflow usability features
- diary improvements should emphasize decision synthesis and handoff quality
- semantic retrieval must remain labeled as research-first on the pre-`1.0` path
