# Haxaml `0.7.x` Roadmap

`0.7.x` is the setup and onboarding line.

`0.6.x` made Haxaml coherent as a governed workflow and MCP runtime. `0.7.0` is the release where Haxaml stops being "good if you already know how to wire it" and becomes a product that can install itself into real agent environments with explicit support policy, drift detection, and documented manual fallbacks.

`0.7.0` is the primary executable release in this line. Later `0.7.x` work exists to deepen automation, close target-specific gaps, and harden upgrade and drift behavior without changing the direction set here.

## Research Method

The target matrix and onboarding policy below were verified against official vendor docs on May 7, 2026.

The `0.7.x` line optimizes for real Haxaml integration viability, not raw brand popularity. A target makes the list only if it has enough documented instructions, rules, skills, subagents, or MCP surfaces for Haxaml to support responsibly.

Two notable exclusions are intentional:

- Roo Code is excluded from the `0.7.x` target list because its official docs announce shutdown on May 15, 2026.
- Aider is excluded from first-class onboarding because it has useful conventions and config surfaces, but not a first-class MCP plus skill onboarding model that matches the `0.7.0` product goal.

## `0.7.0` Product Position

`0.7.0` is the setup and onboarding release.

This is the point where Haxaml becomes a complete install flow instead of a package set that still expects users to know which files each agent reads, where MCP belongs, and which paths are safe to automate.

The onboarding model is two-layer:

- shared Haxaml core: `.haxaml/` plus one canonical Haxaml operating contract
- target-native adapters: rules, AGENTS files, skills, subagents, and MCP config in the target's own format

## `0.7.0` Deliverables

- `haxaml setup` as the main onboarding command
- `haxaml setup doctor` to audit installs, detect drift, and report missing manual follow-up
- `haxaml setup print` to render instruction, skill, agent, or MCP snippets without writing files
- one shared internal target registry consumed by both CLI behavior and docs
- one canonical Haxaml skill recipe rendered into target-specific outputs
- explicit project-scope versus user-scope behavior

## CLI Spec

### `haxaml setup`

Defaults:

- `--scope project`
- `--target auto`
- write missing files
- never overwrite existing files without `--force`
- print a created / skipped / manual-actions summary

Behavior:

- `--scope project` writes only repo files meant to be shared and version-controlled.
- `--scope user` writes only documented home-directory paths and never stores secrets in repo files.
- `--target auto` detects likely agents from existing repo files and installed commands.
- If no target is detected, setup still writes Haxaml's generic repo surfaces: `.haxaml/`, a root `AGENTS.md` adapter, and `.agents/skills/haxaml/SKILL.md`.
- `--only` supports `frame`, `instructions`, `skills`, `agents`, `mcp`.
- `--format` supports `text` and `json`.
- For settings-UI-only or non-stable surfaces, setup must not guess hidden paths. It prints the exact text to apply plus the official docs link instead.

### `haxaml setup doctor`

`setup doctor` must answer three questions clearly:

- what is installed
- what drifted from Haxaml-managed outputs
- what still requires manual action because the target surface is not safely writable

### `haxaml setup print`

`setup print` exists so users and CI can inspect exact adapter output before any write. It is also the fallback surface for targets where Haxaml intentionally refuses to guess undocumented config paths.

## Internal Architecture

`0.7.0` should introduce one shared target registry. Both CLI setup behavior and documentation should read from the same source of truth.

The registry must carry:

- target id and display name
- official docs URLs
- project instruction paths and user instruction paths
- project MCP paths and user MCP paths
- project skill paths and user skill paths
- project agent or subagent paths and user agent or subagent paths
- capability flags
- support tier
- notes for unsupported or manual-only surfaces

`0.7.0` should also keep one canonical Haxaml skill recipe and render target-specific outputs from it. Do not use symlinks in `0.7.0`.

Every generated file should embed Haxaml version, target id, and generator metadata so `setup doctor` can detect drift reliably.

Write policy must be explicit:

- missing files: create
- generated files previously owned by Haxaml: update
- user-authored files with no Haxaml markers: skip unless `--force`
- unsupported surfaces: print manual step and do not mutate

## Per-Target Onboarding Policy

### Claude Code

- Automatically write project `CLAUDE.md`, `.claude/agents/`, and `.mcp.json` when requested.
- Automatically write user `~/.claude/CLAUDE.md`, `~/.claude/agents/`, and `~/.claude.json` when `--scope user` is requested.
- Keep skills out of scope because Claude Code's documented extensibility model is memory, subagents, plugins, and MCP rather than a native checked-in skill format.

### Codex CLI

- Automatically write project `AGENTS.md`, `AGENTS.override.md` only where explicitly requested, and project `.agents/skills/`.
- Automatically write user `~/.codex/AGENTS.md`, `~/.codex/AGENTS.override.md`, `~/.agents/skills/`, and user `~/.codex/config.toml` updates for MCP/config onboarding.
- Treat project-local Codex MCP support as guidance-only in `0.7.0` unless official docs become explicit enough to register a stable repo-local path and schema.

### Cursor

- Automatically write project `.cursor/rules/*.mdc`, project `AGENTS.md`, and project `.cursor/mcp.json`.
- Automatically write user `~/.cursor/mcp.json` when `--scope user` is requested.
- Print guidance for user rules because official docs route that surface through Cursor Settings rather than a stable file path.

### Windsurf

- Automatically write project `.windsurf/rules/*.md`, `AGENTS.md`, and `.windsurf/skills/`.
- Automatically write user `~/.codeium/windsurf/memories/global_rules.md`, `~/.codeium/windsurf/skills/`, and `~/.codeium/windsurf/mcp_config.json`.
- Treat project-local MCP as guidance-only because official docs document user-level MCP config, not a stable repo-local MCP file.

### GitHub Copilot

- Automatically write project `.github/copilot-instructions.md`, `.github/instructions/**/*.instructions.md`, `.github/skills/`, `.github/agents/*.agent.md`, and `.github/mcp.json`.
- Automatically write user `~/.copilot/copilot-instructions.md`, `~/.copilot/instructions/`, `~/.copilot/skills/`, `~/.copilot/agents/`, and `~/.copilot/mcp-config.json`.
- Keep `AGENTS.md` as a cross-agent adapter where helpful, but prefer documented Copilot-native surfaces first.

### Gemini CLI

- Automatically write project `GEMINI.md`, `.gemini/settings.json`, and `.gemini/skills/`.
- Automatically write user `~/.gemini/settings.json` and `~/.gemini/skills/`.
- Treat `GEMINI.md` as project memory and `settings.json` as the main config plus MCP surface.

### Continue

- Automatically write project `.continue/rules/`, `.continue/config.yaml`, and `.continue/mcpServers/` material where the config shape is documented.
- Automatically write user `~/.continue/config.yaml` and `~/.continue/rules/`.
- Do not invent a pseudo-skill format. Continue support in `0.7.0` is rule-first and config-first.

### Cline

- Automatically write project `.clinerules/` and `.cline/skills/`.
- Automatically write user `~/Documents/Cline/Rules` or `~/Cline/Rules`, `~/.cline/skills/`, and `~/.cline/data/settings/cline_mcp_settings.json`.
- Treat project-local MCP as guidance-only because official docs do not establish a stable project file for it.

### OpenCode

- Automatically write project `AGENTS.md` and `.opencode/skills/`.
- Automatically write user `~/.config/opencode/AGENTS.md` and `~/.config/opencode/skills/`.
- Gate MCP config generation behind confirmed registry knowledge of the official file path and schema for `opencode.json` or `opencode.jsonc`.

### Junie

- Automatically write project `.junie/AGENTS.md`, `.junie/skills/`, `.junie/agents/`, and `.junie/mcp/mcp.json`.
- Automatically write user `~/.junie/skills/`, `~/.junie/agents/`, and `~/.junie/mcp/mcp.json`.
- Prefer `.junie/AGENTS.md` as the main project guidance path rather than a repo-root generic instruction file.

## `0.7.1+` Follow-Ons

Later `0.7.x` work should deepen the same setup model instead of changing it.

Planned follow-ons:

- enterprise and system-level installs
- deeper partial-target automation where `0.7.0` deliberately prints guidance
- stronger drift and version enforcement across installed adapters
- broader fixture coverage as target-specific support matures

## Out Of Scope

The `0.7.x` line should stay disciplined about what setup is not:

- no dashboard editing
- no secret manager
- no UI automation of settings dialogs
- no attempt to normalize undocumented private config paths

## Tests And Acceptance Criteria

### Registry tests

- every target has explicit project and user capability flags
- every documented path has a source URL in the registry
- unsupported surfaces are marked manual-only and never silently omitted

### Setup behavior tests

- clean repo plus `haxaml setup` writes `.haxaml/` and generic cross-agent assets
- `--scope project` never writes into `$HOME`
- `--scope user` never writes into the repo
- `--dry-run` produces the same report without writes
- `--force` updates only Haxaml-managed targets, not unrelated user files

### Target fixture tests

- Claude, Cursor, Copilot, Gemini, and Junie fixtures cover full project and user paths
- Windsurf, Continue, Cline, OpenCode, and Codex fixtures cover manual-only gaps correctly

### Doctor tests

- missing skill or MCP config is reported clearly
- version drift in generated files is reported clearly
- unsupported-but-requested surfaces are reported as manual follow-up, not success

### Docs acceptance

- `0.7.x.md` includes direct links to the vendor docs used for each target
- every path in the matrix matches the target registry used by `haxaml setup`

## Assumptions And Defaults

- `0.7.x.md` is the roadmap for the full `0.7` line, not just a single release note.
- `0.7.0` supports all 10 researched targets, but not every target gets the same automation depth.
- Haxaml should prefer official, documented, file-backed surfaces over UI-only configuration.
- When docs are weak or ambiguous, `0.7.0` should choose "print guidance, do not guess."
