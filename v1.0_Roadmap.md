# Haxaml v1.0 Roadmap

This roadmap is the stable-kernel path for Haxaml.

The canonical product direction lives in [goals.md](./goals.md). This file keeps the 1.0 blocker list and release path.

## Current Reality After The 2026-05-17 Audit

Haxaml has a strong model, but the engine is not strict enough yet.

What works:

- FRAME gives the project a structured memory model.
- MCP tools expose a lifecycle instead of one giant prompt.
- Prebuild exists as the architect phase.
- Context packs keep normal work smaller than full repo dumps.
- Archive retrieval exists for follow-up memory.
- Setup has moved toward provider-aware onboarding.

What is not 1.0-ready:

- Hot `acts.yaml` can still bloat even after compaction.
- Archive is lossless but not storage-efficient enough.
- Setup target selection can confuse weak evidence with real selection.
- Setup reruns are not guaranteed safe enough around active FRAME state.
- Prebuild questions and materials are still too advisory.
- Verification can still rely too much on agent claims.
- Tests cover broad surfaces but not enough messy real-world failure modes.

The path to 1.0 is no longer "add more features." It is "remove bypasses."

## Non-Negotiable 1.0 Blockers

These must be closed before Haxaml can call itself stable.

- Hot state must obey a real context budget.
- Archive must preserve full history without keeping bulky replay data in the active loadout.
- Setup must be idempotent by default.
- Provider selection must be explicit, preview-first, and impossible to silently downgrade to generic.
- Prebuild must block unresolved user questions and missing materials.
- Context pack must issue a build permit only after gates pass.
- Verification must prove work against the original plan.
- Tests must reproduce the user-reported bugs and similar edge cases.

## `0.6.x` Foundation

The `0.6.x` line made Haxaml coherent enough to build on.

- `0.6.0`: introduced prebuild, FRAME normalization, semantic validation, and deterministic export recipes.
- `0.6.1`: aligned package/runtime versioning and release helpers.
- `0.6.2`: unified the public lifecycle around `about -> guidance -> prebuild -> context_pack -> verify -> record -> expect_sync`.
- `0.6.3`: reduced response noise and improved context refresh behavior.
- `0.6.4`: added stronger deterministic consistency signals across health, doctor, reconcile, and prebuild.
- `0.6.5`: tightened public docs and packaging.
- `0.6.6`: added Acts archive and governed follow-up retrieval.
- `0.6.7`: added runtime snapshot caching, refresh deltas, and the first read-only local dashboard.

The lesson from `0.6.x`: Haxaml has enough surfaces. The next work is making the surfaces harder to misuse.

## `0.7.x` - Setup, State, And Workflow Trust

Goal: make Haxaml trustworthy where agents already live.

The detailed release plan lives in [`0.7.x_Roadmap.md`](./0.7.x_Roadmap.md). This roadmap keeps the high-level 1.0 view.

Release intent:

- `0.7.0`: setup foundation.
- `0.7.1`: workflow accommodation foundation.
- `0.7.2`: provider workflow depth.
- `0.7.3`: workflow doctor and drift audit.
- `0.7.4`: interactive setup, provider-aware skills, safe config merges, and preview-first onboarding output.
- `0.7.5`: materials, context discipline, handoff quality, state bloat correction, setup idempotency, and lifecycle gate hardening.
- `0.7.6`: real setup TUI correction, setup trust cleanup, and release-pipeline hardening.
- `0.7.7`: `0.7.x` close-out, whole-line retrospective, and explicit handoff into `0.8.0`.

`0.7.4` remains the setup UX release, but the audit found that setup cannot be considered truly done until target selection and rerun safety are proven. `0.7.5` should therefore absorb the practical cleanup needed to protect the README promise.

`0.7.7` should close the line in plain language: setup is now scaffold-like enough to trust, governed state is compact enough to stay usable, and the remaining enforcement-first protocol work belongs to `0.8.0`, not to another drifting `0.7.x` patch.

The close-out artifact lives in [`docs/reports/haxaml-0.7x-closeout.md`](./docs/reports/haxaml-0.7x-closeout.md). Treat it as the handoff note from the setup/state/workflow trust line into the FRAME/protocol maturation line.

### `0.7.4` Exit Criteria

- Interactive setup is usable in a real TTY.
- `--non-interactive` and `--format json` remain machine-safe.
- Shared weak evidence like `AGENTS.md` never silently selects multiple providers.
- Explicit Codex selection installs Codex-specific outputs.
- Generated `SKILL.md` files use valid YAML frontmatter.
- Config merges touch only known Haxaml-owned subtrees.
- Dry-run and apply output show exact paths and previews.

### `0.7.5` Exit Criteria

- Setup rerun over an active project is safe by default.
- FRAME files are create-missing or schema-repair only.
- Hot Acts compaction is budget-aware, not just count-aware.
- `completed_tasks` no longer grows forever in hot state.
- Required prebuild questions and materials become hard gates.
- Context pack returns a build permit when gates pass.
- Verification checks evidence against planned scope.
- Tests cover the user-reported setup, state, and lifecycle failures.

## `0.8.0` - FRAME And Governance Maturation

Goal: mature the FRAME model and protocol while turning Haxaml from guidance-first into enforcement-first.

Planned work:

- Evolve `expect.yaml` from a runbook-only sync target into the high-level governed project checklist for the whole goal.
- Make `expect.yaml` describe the expected run sequence, likely number of runs, and intended outcomes by run, similar to a governed `plan.md` without requiring agent plan mode.
- Keep `expect.yaml` predictive and high-level rather than rigid: the sequence should guide execution, handoff, and verification without pretending every run will unfold exactly as written.
- Define clearer semantics for what belongs in `facts`, `rules`, `acts`, `expect`, and `map` so the FRAME protocol is easier to teach, validate, and evolve.
- Research and document the FRAME protocol itself as a first-class product surface, not just an implementation detail behind the tools.
- Add a first-class Blueprint or PrebuildGate object.
- Add structured user question records and answer status.
- Add materials receipts without storing secrets.
- Block context pack when required gate items are unresolved.
- Compare changed files to planned scope during verification.
- Reject record when verification is stale, failed, missing, or unrelated.
- Add migration helpers for older FRAME projects.

Exit criteria:

- `expect.yaml` can act as the project-level run guideline for the goal, with clear predicted runs and intended outcomes.
- Agents can use `expect.yaml` as a high-level checklist and handoff guide without treating it as a brittle exact-order script.
- FRAME role boundaries are clearer, documented, and migration-safe enough to keep evolving.
- The FRAME protocol is mature enough that future tooling can target it deliberately.
- Agents receive a clear stop response when user input is required.
- Missing materials cannot be ignored in the happy path.
- "No evidence, no done" is enforced by code, not only by prompt text.

## `0.9.0` - Evaluation Readiness

Goal: prove Haxaml works under messy real-world conditions.

Planned work:

- MCP stdio/client contract tests.
- Setup fixture matrix across providers, scopes, modes, and existing file shapes.
- State/archive durability tests with large fixtures, corrupt files, and multiprocess writes.
- Lifecycle state-machine tests for invalid order, restarts, retries, and blocked gates.
- Golden docs examples and CLI transcripts.
- RAG research remains evidence-gated, not a default rollout promise.
- Embedding and meaning-vector experiments should start with typed FRAME roles, not whole-repo brute force.
- Native acceleration investigation should identify hot paths before any Rust or C++ work is approved.

Exit criteria:

- Haxaml is credible for serious evaluation in real repos.
- Every release-critical behavior has a regression test.
- Retrieval and embedding experiments have explicit evidence for or against default adoption.

## RAG And Embeddings Research

Goal: find out whether Haxaml should move from pure text retrieval to typed semantic retrieval.

The core idea is simple:

- text can be embedded into vectors that approximate meaning
- similar meanings end up near each other in vector space
- retrieval can then fetch likely relevant project memory before the agent reads the full source

For Haxaml, the interesting question is not "can we embed text." The interesting question is "which FRAME objects should be embedded, and which should stay structured?"

Candidate FRAME embeddings:

- `facts.yaml`: project facts, stack, constraints, operating assumptions
- `rules.yaml`: policy, mandatory behaviors, setup limits, lifecycle rules
- `acts.yaml`: decisions, runs, session summaries, verification summaries
- `map.yaml`: module ownership and impact relationships
- `expect.yaml`: upcoming objectives and planned outcomes

Research rule:

- embeddings should complement FRAME structure, not replace it
- retrieval should stay evidence-aware, meaning every returned chunk needs a source path and role
- typed retrieval should prefer the right category first, then use similarity inside that category
- if retrieval cannot explain why something was selected, it is not ready for default use

Useful evaluation questions:

- Does semantic retrieval improve task completion on real repo histories?
- Does it reduce context size without hiding critical constraints?
- Does it preserve provenance well enough for verification and record?
- Does it outperform plain keyword or path-based retrieval for the same budget?

## Native Acceleration Path

Goal: only move hot paths into Rust or C++ when measurements justify the complexity.

Haxaml should stay Python-first unless profiling proves a bottleneck. The likely native candidates are:

- vector indexing and nearest-neighbor search
- archive/index compaction
- repeated YAML/JSON serialization on large state files
- token counting or chunk ranking if it becomes a measurable bottleneck
- context pack assembly if Python overhead shows up in real traces

Decision rule:

- use Python while the code is still mostly orchestration and policy
- use Rust first when Haxaml owns the algorithm and needs safety plus packaging control
- use C++ only when an existing library is clearly the best implementation, such as a mature vector backend
- do not introduce native code just because it sounds faster; introduce it when profiling, benchmarks, and maintenance cost all agree

Why Rust first:

- lower memory-safety risk than C++
- easier fit for a governance-focused tool that needs predictable behavior
- good path for PyO3 or a small standalone service when Haxaml needs owned acceleration

When C++ is still reasonable:

- wrapping a best-in-class external ANN or similarity engine
- reusing a mature library where Rust would duplicate too much work
- staying compatible with existing ecosystem tooling that is already C++-anchored

## `1.0.0` - Stable Governance Kernel

Goal: declare Haxaml stable as a governed workflow layer for software repositories.

Stable by 1.0 means:

- FRAME semantics are documented and migration-safe.
- MCP tool names and payload contracts are stable.
- Setup is idempotent and provider-correct.
- Hot state stays small without losing cold history.
- Prebuild blocks missing user input and missing materials.
- Context pack is a scoped build permit, not just a context dump.
- Verification is evidence-bound.
- Record and expect sync preserve handoff quality.
- Optional retrieval acceleration may use typed embeddings or a native backend, but the default path is still governed and explainable.
- The dashboard remains read-only unless a governed write model exists.
- Release checks prevent package, docs, setup, and lifecycle drift.

Enterprise-facing bar for 1.0:

- enough policy, workflow, context, setup, and retrieval maturity for serious adoption review
- not a promise of full org-admin policy distribution, fleet orchestration, or central operations tooling

## 1.0 Go / No-Go Test

Before tagging 1.0, run this plain-language check:

- Can a user run setup twice without losing `.haxaml` state?
- Can the user see exactly which provider is being installed?
- Does selected Codex actually install Codex behavior?
- Does hot `acts.yaml` stay small after repeated work?
- Does Haxaml stop and ask when required information is missing?
- Does Haxaml refuse to record success without evidence?
- Can the next agent continue without reading the whole repo?

If the answer to any item is no, Haxaml is not yet a stable governance kernel.
