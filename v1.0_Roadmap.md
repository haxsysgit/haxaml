# Haxaml v1.0 Roadmap

This roadmap is the consolidated source of truth for what `0.6.x` achieved and what still needs to happen before `1.0`.

The product direction is explicit:

> Haxaml is workflow governance for AI coding agents.

It is not primarily a memory tool. Memory, archive retrieval, context packing, and the local dashboard exist to support governed software work, not to become the product headline.

## What `0.6.x` Actually Delivered

The `0.6.x` line turned Haxaml from an MCP-first governance idea into a more coherent product surface.

### `0.6.0` - Prebuild And FRAME Normalization

- Introduced `haxaml_prebuild` as the main governed entry point.
- Added the normalized in-memory `FrameModel`.
- Added semantic validation and stronger readiness checks.
- Refactored export generation around a deterministic recipe.

### `0.6.1` - Versioning Discipline

- Aligned package/runtime versioning.
- Added centralized release-version helpers.
- Added scripted version bump support.

### `0.6.2` - Lifecycle Unification

- Standardized the public governed flow around:
  `about -> guidance -> prebuild -> context_pack -> verify -> record -> expect_sync`
- Reduced mixed messaging from older session-first public guidance.

### `0.6.3` - Context And Response Noise Reduction

- Tightened repeated `context_pack` policy.
- Reduced lifecycle verbosity.
- Improved utility-mode boundaries and benchmark realism.

### `0.6.4` - Deterministic Consistency Signals

- Added stronger deterministic progress, drift, and consistency checks across validation, reconcile, doctor, health, and prebuild.

### `0.6.5` - Release And Docs Alignment

- Shipped a tighter public docs and package story around MCP-first governance.

### `0.6.6` - Archive And Follow-Up Retrieval

- Added tiered acts archiving.
- Added governed archive retrieval through `haxaml_context_fetch`.
- Kept hot state smaller without losing decision history.

### `0.6.7` - Incremental Reads And First Local Dashboard

- Added a shared in-process runtime snapshot cache for FRAME and archive reads.
- Made repeated `context_pack` calls return smaller refresh deltas with explicit change metadata.
- Added the first local read-only dashboard behind `haxaml dashboard`.
- Split packaging cleanly across `haxaml`, `haxaml-mcp`, and `haxaml-ui`.

## Current Product Reality

Haxaml already does these things well:

- governs agent entry into project work through `about`, `guidance`, and `prebuild`
- normalizes FRAME into a reusable internal model
- validates semantic quality, not just YAML shape
- keeps context pulls smaller than full-file dumps
- preserves governed history through archive and retrieval
- separates core, MCP runtime, and dashboard concerns at packaging level
- setup is shipped; workflow-native accommodation is not

Haxaml is still incomplete in these areas:

- workflow-native accommodation is not yet first-class across real agent runtimes
- proactive lifecycle steering is still weaker than the desired "governor" model
- materials gathering exists, but owner-provided dependencies and `materials_needed` are not prominent enough in live workflow outputs
- context-budget work is useful, but still not productized enough as workflow usability
- diary history still needs higher-level decision synthesis and failure summaries
- semantic retrieval is a moat candidate, but should remain evidence-gated before default adoption

The remaining pre-`1.0` gap is proactive lifecycle steering, not basic onboarding. Haxaml still needs to actively shape real workflows, then harden governance so agents cannot casually bypass planning, materials checks, verification order, or drift boundaries.

Enterprise readiness before `1.0` is mixed by design:

- enough for serious workflow and policy evaluation
- not yet full organization-admin, fleet rollout, or system-wide deployment depth

## `0.7.x` - Setup Foundation And Workflow Accommodation

Goal: turn shipped setup into active, tool-specific workflow accommodation.

`0.7.0` already delivered the setup foundation. The rest of `0.7.x` extends that foundation into hooks, custom agents, background environments, portable skills, workflow doctoring, an interactive setup UX, materials visibility, context-budget discipline, and handoff quality.

The detailed roadmap for this line lives in [`0.7.x_Roadmap.md`](./0.7.x_Roadmap.md).

Core outcomes for the full `0.7.x` line:

- keep `haxaml setup` as the baseline install path, then extend it with workflow-native outputs
- support tool-specific workflow assets without introducing a parallel "workflow MCP tool"
- make Haxaml more proactive inside real agent workflows while keeping `0.7.x` guidance-first
- make materials gathering, context-budget planning, and decision synthesis more visible in everyday use

Planned work:

- `0.7.1`: workflow accommodation foundation, including `workflow` setup assets and a shell-safe workflow guard integration point
- `0.7.2`: provider workflow depth for Claude, Copilot, Cursor, OpenCode, Junie, and Codex-adjacent automation integration points
- `0.7.3`: advisory-first, provider-aware workflow doctor and drift audit for the shipped workflow-capable targets
- `0.7.4`: interactive setup, provider-aware skills, safe config merges, and preview-first onboarding output
- `0.7.5`: materials visibility, context discipline, and diary readability / cross-provider handoff quality

Exit criteria:

- supported targets can install Haxaml and also place it inside their real workflow-native integration points
- workflow accommodation routes into the existing governed lifecycle instead of bypassing it
- materials gathering and context-budget behavior are visibly useful, not just internally correct

## `0.8.0` - Governance Hardening

Goal: harden governance after workflow accommodation exists.

Planned work:

- stronger enforcement when required materials or owner-provided dependencies are missing
- stricter lifecycle ordering so skipped verify or skipped record sequencing is surfaced and blocked reliably
- stronger drift enforcement against lifecycle state, rules, and architectural boundaries
- better done checks before record
- migration helpers for older FRAME projects

This is the release where Haxaml should move more decisively from reactive guidance to proactive enforcement.

Exit criteria:

- recording success without evidence is blocked reliably
- missing `materials_needed` cannot be quietly ignored in the happy path
- common lifecycle drift and verify/record mis-ordering are surfaced before state is recorded

## `0.9.0` - Enterprise Evaluation Readiness And Semantic Retrieval Research

Goal: reach evaluation-grade confidence for serious adoption review while keeping semantic retrieval evidence-gated.

Planned work:

- semantic retrieval research as a first-class track:
  - evals for quality, relevance, and failure modes
  - cost and latency thresholds
  - prototype retrieval paths and archive-integration options
  - explicit adoption criteria for any default rollout decision
- richer adapter and provider conformance tests
- workflow and policy fixture coverage across more real repo shapes
- documentation and demos that show cross-provider governed workflows in practice
- public contract tightening ahead of `1.0`

Semantic retrieval belongs on the pre-`1.0` path because it is a credible moat candidate, but `0.9.0` should treat it as research-first, not as a guaranteed default ship target.

Exit criteria:

- Haxaml is credible for enterprise evaluation of workflow, policy, and retrieval maturity
- semantic retrieval has explicit evidence for or against default adoption

## `1.0.0` - Stable Governance Kernel

Goal: declare Haxaml stable as a governed workflow layer for real software repositories.

Stable by `1.0` means:

- FRAME semantics are documented and compatibility policy is clear
- MCP tool names and payload contracts are stable
- setup, workflow accommodation, verify, and record flows are coherent
- release checks prevent broken package alignment and broken public docs
- a real repo can adopt Haxaml, run it through MCP and workflow-native integration points, inspect it through the dashboard, and upgrade it without hidden tribal knowledge

Enterprise-facing bar for `1.0`:

- enough policy, workflow, and retrieval maturity for serious enterprise adoption review
- not a promise of full org-admin policy distribution, system-wide deployment orchestration, or central operations tooling
