# Haxaml v1.0 Roadmap

This roadmap covers the path from `v0.5.2` to `v1.0`. It merges the engineering requirements with the product direction: making Haxaml a governed workflow controller, not a memory tool. The public lifecycle story is `about → guidance → prebuild → build → verify → record`.

## v0.6.0 - Prebuild Tool And FRAME Schema Engine

Goal: introduce `haxaml_prebuild` as the flagship feature and make FRAME meaning stricter.

Planned work:

- Add `haxaml_prebuild` MCP tool (new `tools_prebuild.py`):
  - produces a structured readiness report before implementation
  - calls `session_start` and `session_plan` internally
  - returns: `readiness_status`, `required_questions`, `materials_needed`, `done_criteria`, `likely_impact`, `risks`, `session_id`
- Add 7 readiness statuses: `ready_to_build`, `needs_user_input`, `needs_project_inspection`, `blocked_by_missing_context`, `utility_mode`, `blocked_by_conflict`, `blocked_by_policy`
- Add 12 task type templates with required questions and done criteria per type: `authentication`, `payment`, `api_endpoint`, `database_migration`, `bug_fix`, `refactor`, `testing`, `deployment`, `documentation`, `cli_command`, `mcp_integration`, `frontend_feature`
- Build a normalized internal FRAME model consumed by validation, context packs, exports, reconcile, and MCP tools.
- Strengthen semantic validation beyond YAML/schema shape:
  - missing operational facts
  - weak or ambiguous rules
  - malformed lifecycle policy
  - missing verification policy
  - stale active task state
  - unclear module ownership in `map.yaml`
- Add doctor checks for quality gaps that are not hard schema failures.
- Refactor export generation around a prompt recipe:
  - role and operating contract
  - project facts
  - non-negotiable rules
  - MCP tool workflow
  - context policy
  - verification policy
  - stop/failure conditions
  - output style
- Create golden tests for exported prompts across supported agent targets.
- Remove deprecated wrappers: `haxaml_run`, `haxaml_done`, `haxaml_context`.

Exit criteria:

- `haxaml_prebuild` returns a structured readiness report. Deprecated wrappers are gone.
- All export targets are generated from the same recipe model.
- Schema/doctor checks catch weak FRAME files before agents depend on them.

## v0.7.0 - Haxaml Skill, Context Budget, And Impact Preview

Goal: ship the complete four-layer product stack and make context selection measurable.

The product stack this milestone delivers:

```
AGENTS.md adapter   = short pointer to Haxaml + Skill + MCP
Haxaml Skill        = token-efficient operating manual
Haxaml MCP          = runtime engine
FRAME               = source of project state
```

Planned work:

**Haxaml Skill system:**

- Add `haxaml/skills/haxaml/SKILL.md` — built-in canonical Skill with: public lifecycle, MCP tool mapping, token policy, prebuild discipline, build discipline, verify discipline, record discipline, and provider-agnostic rule
- Add `haxaml/adapters/` — generated adapter templates for all supported targets, all produced from the same internal recipe (role, lifecycle, MCP mapping, token policy, discipline sections, provider-specific notes)
- Add `haxaml/setup/` package: `detect.py`, `install.py`, `targets/` (one module per agent target)
- Add `haxaml setup` CLI command with subcommands: `scope project`, `scope user`, `target <name>`, `print skill`, `doctor`, `dry-run`, `force`
- Project-scope adapters for 10 targets: Codex, Claude Code, OpenCode, Cursor, Windsurf, GitHub Copilot, Gemini CLI, Cline, Roo Code, Continue
- User-scope setup for targets with stable writable paths; print guidance for all others
- Setup output summary: project root, FRAME status, MCP guidance, Skill status, detected targets, public lifecycle, next steps
- Setup safety: never silently overwrite user-owned files; use `BEGIN HAXAML MANAGED SECTION` / `END HAXAML MANAGED SECTION` markers; write `.haxaml-suggested` file when unsafe to write directly
- Skill versioning: `version:` field in `SKILL.md` header; doctor warns when package version and Skill version drift
- Setup doctor checks: FRAME files exist, Skill installed and versioned, adapter files present, Skill contains token policy and prebuild terminology and MCP tool mapping

**Context budget and impact:**

- Add a context budget planner that selects packs based on task type, risk, and model window budget
- Strengthen one-context-pack-per-session behavior with clearer refresh categories
- Add context diff summaries so refreshes can explain what changed
- Add payload-size and section-omission metadata to every context output
- Add tests for token ceilings, omitted sections, and repeated-call gate behavior
- Add `haxaml_impact_preview` standalone tool — task-level impact preview with `avoid_touching` boundaries, working without `map.yaml` through keyword fallback
- Add `haxaml_done_check` — pre-record gate checking done criteria satisfaction
- Tighten visibility call policy in `about` payload: max one visibility call per session unless a gate fails

Exit criteria:

- `haxaml setup` runs in a clean project and produces FRAME files, Skill, and at least one adapter without error
- All 10 project-scope adapter files generated from the same recipe
- Setup doctor passes on a correctly set-up project
- Agents receive enough context without defaulting to full-project dumps
- Context bloat regressions fail tests
- Impact preview usable without `map.yaml`

## v0.8.0 - Verification Gate And Reconcile Hardening

Goal: make governed execution safer and make "no evidence, no done" a real enforcement boundary.

Planned work:

- Stricter `haxaml_session_verify` — blocks record when `changed_files` is empty but git detects code changes.
- Add `verification_evidence_missing` structured error with a list of missing fields.
- Expand reconcile checks across `facts.yaml`, `rules.yaml`, `acts.yaml`, `expect.yaml`, and `map.yaml`.
- Add impact rules for common project shapes:
  - package/API changes
  - schema changes
  - docs/export changes
  - MCP tool contract changes
  - release/version changes
- Add lifecycle state-machine checks for session start, plan, context, verify, and record.
- Improve utility-task detection so side tasks avoid `.haxaml/*` state writes.
- Add migration helpers for older FRAME files.

Exit criteria:

- Agent cannot record success without supplying `changed_files` and a summary.
- Reconcile catches common cross-file and cross-module drift.
- FRAME migrations preserve compatibility while moving users toward the stricter model.

## v0.9.0 - Prompt Quality, Provider Agnosticism, And Release Candidate

Goal: prove exported prompts are high-quality artifacts and make provider agnosticism visible.

Planned work:

- Add prompt linting for:
  - missing role/task/context/tool/constraint/output sections
  - vague or conflicting instructions
  - missing verification gates
  - unsafe success claims
  - excessive token size
- Add target-specific export conformance tests for Codex, Claude, Cursor, Copilot, Gemini, Windsurf, and generic markdown.
- Add fixture projects that represent real repo patterns and validate exported prompts against them.
- Add provider switch demo documentation showing the same project workflow across Claude Code, Codex CLI, Cursor, Copilot, and Windsurf.
- Extend `haxaml_adopt_plan` with conflict/drift detection between native instruction files (`AGENTS.md` vs `CLAUDE.md` vs `.cursor/rules`).
- Freeze the public MCP tool contract for `1.0` review.
- Produce an upgrade guide from `0.5.x` to the `1.0` release candidate.

Exit criteria:

- Exported prompts pass linting. Provider examples tested.
- Public MCP tool contracts are documented and stable.
- Known migration risks are documented before `1.0`.

## v1.0.0 - Stable Haxaml Core

Goal: declare Haxaml stable for MCP-first agent governance in real repositories.

Planned work:

- Stabilize FRAME file semantics and schema compatibility policy.
- Stabilize MCP tool names, payload contracts, error codes, and lifecycle gates.
- Stabilize export recipe behavior and target-agent profiles.
- Publish final docs for:
  - FRAME model
  - MCP operator workflow
  - prompt compiler recipe
  - schema and validation engine
  - context budget policy
  - migration and compatibility rules
- Add release checks that block publishing when:
  - package versions diverge
  - MCP dependency ranges are misaligned
  - golden exports drift unexpectedly
  - workflow token budgets regress
  - public docs point at missing paths

Exit criteria:

- `haxaml` and `haxaml-mcp` are version-aligned.
- Full test suite passes with schema, export, context, lifecycle, and release checks.
- Docs describe the stable public contract.
- Haxaml can be adopted by a repo, used through MCP, exported to native agent files, and upgraded without relying on hidden project knowledge.
- `haxaml_prebuild` is the documented flagship tool for the prebuild phase.
