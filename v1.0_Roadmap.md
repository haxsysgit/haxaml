# Haxaml v1.0 Roadmap

This roadmap is the consolidated source of truth for what `0.6.x` achieved and what still needs to happen before `1.0`.

The product direction is now explicit:

> Haxaml is prebuild and governance for AI coding agents.

It is not primarily a memory tool. Memory, archive retrieval, context packing, and the local dashboard exist to support governed software work, not to become the product headline.

## What `0.6.x` Actually Delivered

The `0.6.x` line turned Haxaml from an MCP-first governance idea into a more coherent product surface.

### `0.6.0` - Prebuild And FRAME Normalization

- Introduced `haxaml_prebuild` as the main governed entry point.
- Added the normalized in-memory `FrameModel`.
- Added semantic validation and stronger readiness checks.
- Refactored export generation around a deterministic recipe.

### `0.6.1` - Versioning Discipline

- Aligned package/runtime versioning.
- Added centralized release-version helpers.
- Added scripted version bump support.

### `0.6.2` - Lifecycle Unification

- Standardized the public governed flow around:
  `about -> guidance -> prebuild -> context_pack -> verify -> record -> expect_sync`
- Reduced mixed messaging from older session-first public guidance.

### `0.6.3` - Context And Response Noise Reduction

- Tightened repeated `context_pack` policy.
- Reduced lifecycle verbosity.
- Improved utility-mode boundaries and benchmark realism.

### `0.6.4` - Deterministic Consistency Signals

- Added stronger deterministic progress, drift, and consistency checks across validation, reconcile, doctor, health, and prebuild.

### `0.6.5` - Release And Docs Alignment

- Shipped a tighter public docs and package story around MCP-first governance.

### `0.6.6` - Archive And Follow-Up Retrieval

- Added tiered acts archiving.
- Added governed archive retrieval through `haxaml_context_fetch`.
- Kept hot state smaller without losing decision history.

### `0.6.7` - Incremental Reads And First Local Dashboard

- Added a shared in-process runtime snapshot cache for FRAME and archive reads.
- Made repeated `context_pack` calls return smaller refresh deltas with explicit change metadata.
- Added the first local read-only dashboard behind `haxaml dashboard`.
- Split packaging cleanly across `haxaml`, `haxaml-mcp`, and `haxaml-ui`.

## Current Product Reality

Haxaml already does these things well:

- governs agent entry into project work through `about`, `guidance`, and `prebuild`
- normalizes FRAME into a reusable internal model
- validates semantic quality, not just YAML shape
- keeps context pulls smaller than full-file dumps
- preserves governed history through archive + retrieval
- separates core, MCP runtime, and dashboard concerns at packaging level

Haxaml is still incomplete in these areas:

- setup is still manual compared with the intended product vision
- the Skill and adapter layer is not yet first-class
- verification evidence is stricter than before, but not yet a hard product moat
- dashboard is inspect-only and still intentionally thin
- outcome tracking, drift enforcement, and cross-project learning are not finished

## `0.7.0` - Setup, Skill, And Productized Onboarding

Goal: make Haxaml easy to install correctly and hard to use badly.

This is the release where Haxaml becomes a complete setup flow, not just a good MCP package.

### Core outcomes

- Add `haxaml setup` as the main onboarding command.
- Ship a built-in canonical Haxaml Skill.
- Generate or install target-specific adapters from one shared recipe.
- Make package separation operationally clear:
  - `haxaml` = core governance engine and CLI
  - `haxaml-mcp` = MCP runtime
  - `haxaml-ui` = local dashboard
- Keep the dashboard read-only, but integrate it into the setup story.

### Planned work

#### 1. Setup flow

- Add `haxaml setup` with:
  - project-scope setup
  - user-scope guidance where safe
  - dry-run mode
  - force/overwrite controls
  - setup doctor output
- Detect project root, FRAME state, likely agent/editor targets, and MCP readiness.
- Create or validate `.haxaml/` files during setup.

#### 2. Skill system

- Add a canonical built-in Haxaml Skill.
- Define the stable operating contract for:
  - when to call `prebuild`
  - when to call `context_pack`
  - when to use `context_fetch`
  - how to verify before record
  - how to stay token-efficient
- Version the Skill and warn when installed adapters drift from package version.

#### 3. Adapter generation

- Generate adapters for agent environments from one shared recipe.
- Support project-scope adapters first.
- Provide user-scope installation only where paths are stable and safe to write.
- Fallback to printed guidance when automated install would be risky.

#### 4. Context-budget discipline

- Add explicit context budget planning so packs are selected against task/risk/model budget instead of static defaults alone.
- Keep refresh deltas small and measurable.
- Continue strengthening one-pack-first discipline.

#### 5. Impact preview and done checks

- Add a standalone impact preview surface.
- Add a done-check gate before record.
- Strengthen “no evidence, no done” as an actual operational rule.

### Exit criteria

- A clean repo can run `haxaml setup` and get usable FRAME + Skill + MCP guidance without manual spelunking.
- Supported adapters are generated from one source recipe.
- Setup doctor can explain what is missing and what is already correct.
- Context behavior remains token-disciplined under test.

## `0.8.0` - Verification And Drift Enforcement

Goal: make governance harder to bypass once work has started.

Planned work:

- stronger verify/record enforcement
- drift checks against rules and known constraints
- lifecycle state-machine hardening
- impact rules for common change types
- migration helpers for older FRAME projects

Exit criteria:

- recording success without evidence is blocked reliably
- common cross-file drift is surfaced before state is recorded

## `0.9.0` - Export Quality And Provider-Agnostic Confidence

Goal: prove the exported guidance layer is production-grade across agent targets.

Planned work:

- prompt linting
- adapter conformance tests
- richer fixture projects
- provider-switch demos and docs
- public MCP contract freeze for `1.0`

Exit criteria:

- exported prompts/adapters are tested, linted, and stable enough for contract review

## `1.0.0` - Stable Haxaml Core

Goal: declare Haxaml stable as a governed workflow layer for real software repositories.

Stable by `1.0` means:

- FRAME semantics are documented and compatibility policy is clear
- MCP tool names and payload contracts are stable
- setup, skill, adapter, verify, and record flows are coherent
- release checks prevent broken package alignment and broken public docs
- a real repo can adopt Haxaml, run it through MCP, inspect it through the dashboard, and upgrade it without hidden tribal knowledge
