# 0.4 Execution Pack (Roadmap Ops First)

## Scope Lock

- This pack is planning operations only.
- No 0.4.1 feature code starts until review gate closure in `temp/0.4.review.md`.
- All roadmap artifacts stay under `temp/` in this phase.

## Theme Order (Fixed)

`reconcile -> adoption -> docs -> verify depth -> context efficiency -> stability`

Milestones must execute in this exact order: `0.4.1 -> 0.4.2 -> 0.4.3 -> 0.4.4 -> 0.4.5 -> 0.4.6`.

## Release Rule: Commit Message Style

Applies to all 0.4.x patches:
- Use short, accurate, plain-language commit messages.
- Do not use conventional prefixes: `fix:`, `feat:`, `refactor:`, `chore:`, `docs:`.

## Milestone Dependency Graph

- `0.4.1` depends on: review gate closed.
- `0.4.2` depends on: `0.4.1` contract merged and validated.
- `0.4.3` depends on: `0.4.2` adoption outputs and terminology stable.
- `0.4.4` depends on: `0.4.1` reconcile structure and `0.4.2` map/adoption alignment.
- `0.4.5` depends on: `0.4.3` docs examples and `0.4.4` verify signal definitions.
- `0.4.6` depends on: `0.4.1` through `0.4.5` complete and regression-stable.

## Milestone Execution Sequence

## 0.4.1 Reconcile Polish + Better Fix Suggestions

Entry Criteria:
- `temp/0.4.review.md` required-fix checklist is fully closed.
- `uv run pytest -q` passes on the baseline branch before new 0.4.1 changes.
- Contract source-of-truth is `temp/0.4.1.spec.md`.

Definition of Done:
- Every reconcile conflict item includes all required guidance fields.
- `safe_to_auto_apply` is always `false` for 0.4.1 outputs.
- `human_summary` is deterministic for equivalent input state.
- Reconcile tests cover representative blocking + warning conflict combinations.

Acceptance Checks:
- Contract conformance tests verify required fields are always present when conflicts exist.
- Snapshot/structure checks verify deterministic `human_summary` behavior.
- `uv run pytest -q` passes.

Exit Criteria:
- 0.4.1 contract behavior documented and test-backed.
- No open blocking defects in reconcile output structure.

## 0.4.2 Adopt Plan Improvements

Entry Criteria:

- Native-file detection matrix for adoption is agreed.

## 0.4.3 MCP Docs and Example Projects

Entry Criteria:
- 0.4.2 output terms and workflow semantics are stable.

Definition of Done:

- A small end-to-end demo flow is documented.

## 0.4.4 Map-Aware Verification v1

Entry Criteria:

Acceptance Checks:

## 0.4.5 Context Pack Compaction

Entry Criteria:

Acceptance Checks:
- Tests verify stable inclusion behavior for same task input.

Exit Criteria:

## 0.4.6 Dogfooding and Stability

Entry Criteria:
- 0.4.1 through 0.4.5 complete and green.

Definition of Done:
- Dogfood on Haxaml and at least one real project.
- Convert observed failure modes into tests/docs/error-message improvements.
- Changelog/migration notes tightened for 0.4 line stability.

Acceptance Checks:
- Regression tests added for observed real-world issues.
- End-to-end sessions complete with no unresolved blocking defects.
- `uv run pytest -q` passes.

Exit Criteria:
- 0.4 line judged stable enough to begin formal 0.5 direction planning.

## Deterministic Handoff Rules

- No milestone can start before prior milestone exit criteria are explicitly marked complete.
- If acceptance checks fail, milestone status remains in-progress; next milestone is blocked.
- For each milestone handoff, record:
  - completed DoD bullets
  - acceptance check results
  - known residual risks
  - explicit go/no-go for next milestone

## Gate Before Any Future 0.4.1 Coding

Run only after required fixes in review are applied:
- `uv run pytest -q`

If green and review checklist closed, 0.4.1 can move from paused to active.
