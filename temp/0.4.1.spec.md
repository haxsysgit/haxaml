# 0.4.1 Spec: Reconcile Output Contract (Decision-Complete)

Status: planned contract for 0.4.1 implementation.
Scope: `haxaml_reconcile` / `reconcile_derivation` conflict output shape and semantics.

## Contract Goal

Make reconcile output deterministic and directly usable for agent repair workflows by requiring explicit fix guidance on every conflict object.

## Non-Goals

- No runtime API changes are shipped in this roadmap-ops phase.
- This spec does not introduce auto-apply behavior in 0.4.1.

## Conflict Object Contract (0.4.1)

For every entry in `conflicts[]`, the following fields are required and always present:

Existing fields (must remain):
- `id`: stable per-report ordinal format `conflict-###`.
- `category`: conflict class key.
- `severity`: `blocking | warning`.
- `canonical_path`: source-of-truth path.
- `derived_path`: derived/consumer path.
- `canonical_value`: canonical data/value.
- `derived_value`: derived data/value.
- `message`: human-readable mismatch statement.
- `suggested_fix_action`: direct corrective action.

New required guidance fields:
- `fix_confidence`
- `safe_to_auto_apply`
- `related_files`
- `why_it_matters`
- `suggested_next_tool`

## New Field Definitions

### `fix_confidence`

Type: string enum
Allowed values: `low`, `medium`, `high`

Semantics:
- `high`: fix is deterministic, local, and does not require policy interpretation.
- `medium`: fix is likely correct but may require project-specific confirmation.
- `low`: fix is ambiguous, cross-file, or policy-dependent; manual decision required.

### `safe_to_auto_apply`

Type: boolean
Allowed values in 0.4.1: always `false`

Semantics:
- This is a policy lock for 0.4.1.
- Even when `fix_confidence=high`, agents must not auto-apply by policy.

### `related_files`

Type: array of strings
Allowed values: non-empty normalized relative paths (for example `.haxaml/map.yaml`)

Semantics:
- Must include the canonical and derived files implicated by the conflict.
- May include additional directly relevant files only.
- Must not include unrelated files.

### `why_it_matters`

Type: string
Allowed values: free text, minimum one complete sentence

Semantics:
- Explains operational risk of leaving the conflict unresolved.
- Must connect to validation gates, run safety, or boundary integrity.
- Must avoid generic filler.

### `suggested_next_tool`

Type: string enum
Allowed values:
- `haxaml_validate`
- `haxaml_doctor`
- `haxaml_impact`
- `haxaml_session_verify`
- `haxaml_context_pack`
- `manual_edit`

Semantics:
- Recommends the immediate next action after reading the conflict.
- `manual_edit` is used when no tool call can safely resolve the issue.
- Value must align with `fix_confidence` and conflict type.

## Determinism Requirements

- Equivalent repository state must produce equivalent `human_summary` and equivalent conflict field presence.
- Field presence is strict: if `conflicts` contains N items, each item has all required existing and new fields.
- `related_files` ordering must be deterministic (stable sort).

## Human Summary Requirement

`human_summary` must be deterministic and concise:
- Blocking present: include blocking + warning counts.
- Warning-only: include warning count and explicit no-blocking statement.
- No conflicts: explicit healthy-state statement.

## Validation and Test Requirements

0.4.1 implementation must include tests for:
- Required-field presence on every conflict item.
- `safe_to_auto_apply=false` invariant.
- Enum validation for `fix_confidence` and `suggested_next_tool` values.
- Deterministic output behavior for equivalent input state.

## Backward-Compatibility Expectation

- Existing top-level reconcile keys remain available.
- Consumers that only read prior keys continue to work.
- New keys are additive at conflict-item level.

## Implementation Gate

0.4.1 coding may begin only after:
- `temp/0.4.review.md` required-fix checklist is closed.
- Baseline regression check is green via `uv run pytest -q`.
